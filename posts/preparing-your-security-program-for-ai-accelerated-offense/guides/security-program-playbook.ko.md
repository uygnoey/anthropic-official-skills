# AI 공격 가속화 시대를 위한 보안 프로그램 플레이북

Anthropic 보안 엔지니어링 및 연구 팀이 2026년 4월 10일에 게시한 [Preparing your security program for AI-accelerated offense](https://claude.com/blog/preparing-your-security-program-for-ai-accelerated-offense)를 바탕으로 정리한 롤아웃 플레이북입니다. 이 포스트의 지침은 기존 SOC 2 및 ISO 27001 컨트롤에 직접 매핑됩니다. 새로 창작된 내용은 없습니다.

## 배경

AI는 취약점이 발견되고 악용되는 속도를 바꾸고 있습니다. 같은 주 초에 Anthropic은 방어적 사이버보안을 위해 Claude Mythos Preview를 활용하는 Project Glasswing을 발표했습니다. 앞으로 24개월 안에, 어쩌면 수년간 아무도 눈치채지 못했던 수많은 버그들이 AI 모델에 의해 발견되고 연쇄적인 익스플로잇으로 조합될 것입니다. AI 도구를 먼저 도입한 방어자도 더 빠르게 움직일 수 있습니다. 아래는 "지금 당장 해야 할 일" 일곱 가지 영역과 실행 방법, 그리고 보고서 품질 및 소규모 팀을 위한 간소화 방안입니다.

## 1. 패치 격차 해소

AI 모델은 패치되지 않은 시스템에서 이미 패치된 취약점의 특징을 식별하는 데 매우 뛰어납니다. 패치를 역분석해 실동작하는 익스플로잇으로 만드는 과정은 이 모델들이 탁월한 기계적 분석 작업이므로, 패치 공개부터 익스플로잇 등장까지의 창이 좁아지고 있습니다.

- **CISA Known Exploited Vulnerabilities (KEV) catalog**에 있는 모든 항목을 즉시 패치하십시오. 목록에 있는 항목 중 네트워크에서 접근 가능한 것은 긴급 사안으로 처리해야 합니다.
- 나머지 우선순위 결정에는 **EPSS**(Exploit Prediction Scoring System)를 활용하십시오. CVE가 향후 30일 내에 악용될 확률을 매일 업데이트로 제공합니다.
- 인터넷에 노출된 시스템의 패치 소요 시간을 줄이십시오: 익스플로잇 공개 후 **24시간 이내**, 그 외 취약점은 수일 이내.
- 자동화된 업데이트로 인한 중단이 허용 가능한 위험인 경우, 패치 배포와 재부팅을 자동화하십시오. 수동 승인 단계는 지연을 유발하며, **지연이 이제 가장 큰 위험 요소입니다.**

**실용적인 팁.** 대부분의 클라우드 및 OS 벤더는 이미 패치 자동화를 제공합니다. 활성화는 대개 단순한 설정 변경으로 충분합니다. 컨테이너 이미지와 의존성 매니페스트의 경우, 여러 오픈 소스 스캐너가 단일 CI 단계로 실행되며 CVE에 KEV 및 EPSS 데이터를 주석으로 추가하므로 우선순위 지정이 내장되어 있습니다.

→ 재사용 가능한 [closing-the-patch-gap 스킬](../skills/closing-the-patch-gap/SKILL.md)을 참조하십시오.

## 2. 훨씬 더 많은 취약점 보고서를 처리할 준비

취약점 관리 프로세스는 벤더 및 업스트림으로부터 더 많은 패치에 대비해야 합니다.

- **보고량의 수십 배 증가를 계획하십시오.** 보안 회의가 여전히 스프레드시트와 주간 회의 중심이라면 따라가기 어렵습니다.
- 오픈 소스 의존성의 보안을 점검하십시오. **OpenSSF Scorecard**를 사용하면 CI에서 실행되며 관리되지 않는 패키지와 누락된 신호(브랜치 보호, 퍼징 커버리지, 서명된 릴리스, 유지 관리자 활동)를 표시합니다.
- 제3자 위험 관리를 통해 벤더에게도 동일한 기대치를 적용하십시오: 가속화된 익스플로잇 타임라인에 어떻게 대비하고 있는지, 자체 코드를 스캔하고 있는지 물어보십시오.

**실용적인 팁.** 취약한 코드의 도달 가능성을 평가하는 도구를 검토하십시오. 빠른 배포에 대한 신뢰를 얻기 위해 회귀 테스트를 포함해 새로운 업데이트를 지속적으로 제공하는 자동화 프로세스를 구축하십시오.

**취약점 관리에서의 AI 활용 지점:**
- *트리아지 가속화.* 프론티어 모델은 중복 보고서를 제거하고, 자산 정보를 기반으로 노출 범위를 추정하며, 이미 식별된 영향 코드 경로와 함께 교정 티켓 초안을 작성할 수 있습니다.
- *의존성 중복 검사.* 대규모 코드베이스에는 같은 역할을 하는 라이브러리가 여러 개 쌓이는 경향이 있습니다(HTTP 클라이언트 여러 개, JSON 파서 여러 개 등). LLM에 잠금 파일을 가리키고 어떤 의존성이 겹치는지(그리고 통합하면 어떻게 될지) 물어보는 것은 한 시간짜리 작업으로 종종 성과를 냅니다.
- *업그레이드 자동화.* 프론티어 모델은 취약점 보고서와 함께 패치를 생성하고, 테스트하고, 테스트가 통과하는지 검증하면서 업스트림 패치를 수락할 수 있습니다.
- *AI 벤더링.* OpenSSF Scorecard 점수가 낮고 활발히 유지 관리되지 않는 소규모 의존성의 경우, 실제로 사용하는 기능만 LLM이 재구현하도록 하는 것을 고려하십시오.

## 3. 출시 전에 버그 찾기

- **정적 분석 및 AI 보조 코드 리뷰**를 CI에 추가하고, 높은 신뢰도의 발견에 대해 병합을 차단하십시오. 오탐이 이를 비실용적으로 만든다면 게이트가 아닌 도구를 수정하십시오.
- **OWASP ASVS**는 세 가지 엄격성 수준에서 "통과"의 기준을 정의합니다.
- **자동화된 침투 테스트**를 지속적 배포에 추가하십시오. 공격자가 프로덕션에 실행할 것과 동일한 스캔을 스테이징에도 실행하십시오.
- **빌드 파이프라인을 보안화하십시오.** **SLSA** 프레임워크는 등급별 경로를 제공합니다. OpenSSF는 SLSA Level 3 증명을 생성하는 재사용 가능한 GitHub Actions 워크플로를 게시하고 있으며, SLSA 스펙이 시사하는 것보다 훨씬 적은 작업으로 구현할 수 있습니다.
- **Secure by Design** 관행을 최소 기준으로 채택하십시오(CISA의 서약: 기본 MFA, 기본 비밀번호 없음, 투명한 취약점 보고).
- 새 코드에는 **메모리 안전 언어**(Rust, Go, 관리형 런타임)를 선호하십시오. 기존 C/C++ 코드는 재작성이 필요 없지만, 새로운 C/C++ 코드는 정당한 이유가 필요합니다. AI 보조 재작성도 점점 더 실용적인 선택지가 되고 있습니다.

**실용적인 팁.** OWASP Top 10 및 언어별 규칙 세트를 갖춘 SAST 도구는 오픈 소스와 코드 호스팅 플랫폼 내장형 모두 폭넓게 사용 가능합니다. GitHub의 CodeQL이 가장 일반적인 시작점입니다.

**빌드 단계에서의 AI 활용 지점:**
- *AI 취약점 스캔.* 공격자가 사용할 법한 모델로 자신의 코드와 시스템을 먼저 스캔하십시오. 격리된 에이전트, 노이즈 필터링을 위한 검증 단계, 그리고 기존 트리아지 프로세스와의 연결이 필요합니다. **이 섹션에서 한 가지만 구현한다면 이것을 구현하십시오.** → [vulnerability-scanning-agent](../agents/vulnerability-scanning-agent.md)를 참조하십시오.
- *패치 생성.* 프론티어 모델은 일반적으로 SAST 또는 스캐너 발견에 대한 패치 제안이 가능합니다. 개발자의 역할이 "버그를 이해하고 수정 작성"에서 "제안된 수정이 맞는지 검증"으로 이동하며, 후자가 더 빠릅니다. 메모리 안전 마이그레이션에도 동일한 접근 방식이 적용됩니다: 독립적인 C 모듈을 테스트와 함께 Rust로 포팅하고 동등성을 검증합니다.

## 4. 기존 코드에서 취약점 찾기

오랫동안 운영 중인 프로덕션 코드의 대부분은 인간이 여러 번 검토했지만 프론티어 모델로는 한 번도 검사받지 않았습니다. 선제적 스캔은 공격자보다 먼저 현대 LLM의 사정권 안에 있는 취약점을 식별합니다.

- **노출 순서로 우선순위를 정하십시오.** 신뢰할 수 없는 입력을 파싱하거나, 인증 결정을 내리거나, 인터넷에서 도달 가능한 코드부터 시작하십시오.
- **레거시 코드를 포함하십시오.** 현재 검토 관행 이전에 작성된 코드는 최근 검토가 가장 적고, 새로운 점검으로 얻을 것이 가장 많습니다.
- **교정 예산을 확보하십시오.** 오래된 코드의 모델 스캔은 일반적으로 SAST 롤아웃보다 발견 건수가 적지만 실제 취약점 비율이 더 높습니다. 엔지니어링 시간을 계획하십시오.

**실용적인 팁.** 현재 소유자가 적은 인터넷 노출 서비스 하나를 선택해 입력 처리와 인증 로직을 스캔하십시오. 에이전트를 격리 환경에서 실행하고, 검증 단계를 추가하고, 확인된 발견 사항에 조치하십시오. 한 서비스를 제대로 수행하면 더 광범위한 프로그램 비용을 추정하는 합리적인 기반이 됩니다.

## 5. 침해를 전제한 설계

마찰(공격을 지루하게 만드는 것)에서 가치를 얻는 완화책은 끝없는 인내력을 가진 공격자에게는 덜 효과적입니다. 무제한의 공격자 인내력에도 버티는 컨트롤을 선호하십시오.

- **제로 트러스트 아키텍처를 채택하십시오.** 서비스 간 모든 요청을 인터넷에서 온 것처럼 인증하고 권한을 부여하십시오. **CISA의 Zero Trust Maturity Model**과 **NCSC의 zero trust principles** 모두 단계적 도입 경로를 제공합니다.
- **자격 증명이 아닌 검증된 하드웨어에 접근을 연결하십시오.** 프로덕션 시스템과 민감한 내부 도구는 하드웨어 신원이 증명된 관리형 직원 기기에서만 접근 가능해야 하며, **피싱 저항 2FA(FIDO2 / 패스키)**와 함께 사용해야 합니다. 도난된 자격 증명만으로는 충분하지 않아야 합니다. 서비스 간 호출도 하드웨어 신원에 뿌리를 두어야 합니다.
- **서비스를 신원별로 격리하십시오.**
- **장기 비밀을 신원 공급자가 발급한 단기, 좁은 범위의 토큰으로 교체하십시오.** 정적 API 키와 임베드된 자격 증명은 모델 보조 코드 분석을 가진 공격자가 가장 먼저 찾는 것 중 하나입니다.

**실용적인 팁.** 완전한 제로 트러스트는 수년에 걸친 프로그램이지만, **신원 인식 접근 프록시**는 내부 서비스를 재아키텍처링하지 않고도 기기 검증과 MFA 게이팅을 추가합니다. 모든 주요 클라우드 공급자는 기본 옵션과 오픈 소스/상용 대안을 제공합니다. 비밀 관리의 경우, 가장 광범위하게 공유되는 자격 증명 하나를 관리형 시크릿 저장소로 옮기고 교체하십시오. 이것이 나머지를 위한 유용한 강제 함수가 됩니다.

## 6. 노출 범위 축소 및 인벤토리

두 가지 원칙: 알지 못하는 것은 방어할 수 없으며, 노출된 표면이 작을수록 공격 대상도 줄어듭니다.

- 모든 인터넷 노출 호스트, 서비스, API 엔드포인트의 **현행 인벤토리**를 유지하십시오. 침투 테스트와 레드팀 활동에 이를 포함하십시오.
- **사용하지 않는 시스템을 폐기하십시오.** 명확한 소유자가 없는 레거시 서비스는 대개 패치도 되지 않습니다.
- 각 서비스가 노출하는 것을 **최소화하십시오.** 네트워크 인그레스는 기본 거부로; API 표면은 실제로 필요한 것으로 제한하십시오.

**실용적인 팁.** 인터넷 전체 스캔 인덱스는 공개적으로 검색 가능합니다. 자신의 IP 범위와 도메인을 쿼리해 공격자의 정찰이 무엇을 보는지 확인하십시오. 클라우드 자산의 경우, 기본 인벤토리 도구(**AWS Config, Azure Resource Graph, GCP Asset Inventory**)가 이미 존재합니다. 작업은 이를 쿼리하는 것입니다.

**노출 관련 AI 활용 지점:**
- *오래된 코드와 시스템 정리.* 코드베이스와 트래픽 로그에 읽기 접근이 있는 모델은 호출자가 없고 트래픽을 받지 않은 엔드포인트를 나열하고, 각각을 제거하면 어떤 영향이 있는지 설명할 수 있습니다.
- *자율 외부 레드팀.* AI 공격 에이전트를 자격 증명과 소스 접근 없이 외부에서 자신의 경계에 직접 향하게 하십시오. 소스 스캐닝이 놓치는 것을 포착합니다: 잊혀진 호스트, 노출된 관리 인터페이스, 기본 자격 증명, 잘못 구성된 스토리지. 인벤토리 갱신과 동일한 주기로 실행하십시오. → [external-red-team-agent](../agents/external-red-team-agent.md)를 참조하십시오.

## 7. 인시던트 대응 시간 단축

익스플로잇은 패치 후 몇 시간 만에 나타날 수 있습니다. 며칠 단위로 측정되는 대응 프로세스는 너무 느립니다.

- **경보 대기열 앞에 모델을 배치하십시오.** 모든 수신 경보는 사람이 보기 전에 자동화된 초기 조사를 거칩니다. 읽기 전용 SIEM 접근이 있는 "트리아지 에이전트"는 인간의 판단이 가장 필요한 경보에 주의를 집중시킵니다. → [security-triage-agent](../agents/security-triage-agent.md)를 참조하십시오.
- **먼저 체류 시간과 커버리지를 계측하십시오.** 익스플로잇 창이 좁아질 때 가장 중요하며, AI 자동화가 가장 크게 움직일 수 있는 두 가지 지표입니다.
- **인시던트 관련 기록 작업을 자동화하십시오.** 모델이 메모를 작성하고, 아티팩트를 캡처하고, 병렬 조사 트랙을 추구하고, 사후 검토 초안을 작성합니다. **인간**은 봉쇄, 공개, 고객 커뮤니케이션 결정을 내립니다. 인간의 의사 결정 속도가 증거 수집이나 문서 작성으로 인해 제한받아서는 안 됩니다.
- **모델이 탐지 플라이휠을 구동하게 하십시오** — 위협 인텔 수집, 후보 탐지 생성, 헌팅, 발화 조정.
- **다섯 건의 동시 인시던트를 위한 테이블탑을 실시하십시오.** AI 역량을 고려할 때 "월요일의 단일 치명적 CVE" 가정은 현명하지 않을 수 있습니다. 같은 주에 다섯 건의 인시던트로 스트레스 테스트하십시오.
- **MITRE ATT&CK에 대한 탐지 커버리지를 매핑하십시오.** 공격자 기술의 표준 어휘입니다. 무엇을 탐지할 수 있고 없는지 아는 것이 일반적인 "탐지 개선" 목표보다 유용합니다. 횡적 이동과 자격 증명 접근에 대한 커버리지를 우선시하십시오.
- **긴급 변경 절차를 사전에 수립하십시오.** 프로덕션 패치에 대한 2주짜리 변경 승인 주기 자체가 보안 위험입니다. 봉쇄 조치(서비스 오프라인, 자격 증명 교체, 네트워크 경로 차단)도 마찬가지입니다. 누가 어떻게 빠르게 승인하는지 사전에 결정하십시오.

**실용적인 팁.** 알려진 높은 오탐률을 가진 시끄러운 규칙 하나를 선택하십시오. 프론티어 모델을 읽기 전용 접근으로 경보 스트림에 연결하고, 모든 발화에 대해 구조화된 처리 결과를 생성하게 하고, 2주 동안 인간 검토자와의 일치도를 측정하십시오. 일치도가 허용 가능하다면 다음 규칙으로 확장하십시오. 전체 대기열을 한 번에 자동화하려 하지 마십시오. 별도로, **Atomic Red Team**은 ATT&CK 기술에 매핑된 작고 안전한 테스트의 오픈 소스 라이브러리입니다. 몇 가지를 실행하고 기존 로깅이 실제로 감지한 것을 확인하는 것은 반나절짜리 작업으로 구체적인 커버리지 맵을 만들어냅니다.

## 취약점 보고서 제출

코드(자신의 의존성, 오픈 소스 프로젝트, 벤더 제품)를 스캔하고 업스트림에 보고하는 경우, 보고서 품질이 누군가 발견 사항에 조치를 취할지 여부를 결정합니다. 오픈 소스 유지 관리자들은 이미 낮은 품질의 자동화 보고서를 대량으로 받고 있으며, 많은 이들이 AI가 생성한 것처럼 보이는 것은 무시합니다. **보고서는 인간이 검증하고 자신의 이름을 올릴 의향이 있는 경우에만 전송해야 합니다.**

제출 체크리스트와 자가 점검을 위한 재사용 가능한 [writing-quality-vuln-reports 스킬](../skills/writing-quality-vuln-reports/SKILL.md)을 참조하십시오.

## 보안 팀이 없는 경우

위의 대부분의 조언은 전담 보안 기능을 가정합니다. 솔로 개발자, 오픈 소스 유지 관리자, 소규모 조직: 위험은 동일하게 적용되지만 취할 조치는 더 단순합니다.

- **자동 업데이트를 켜십시오** — OS, 브라우저, 제공하는 모든 애플리케이션에 대해. 사용 가능한 단일 가장 효과적인 조치이며 지속적인 노력이 필요 없습니다.
- **자가 호스팅보다 관리형 서비스를 선호하십시오.** 보안 팀이 있는 공급자가 데이터베이스, 인증, 이메일을 운영하도록 하면 패치 부담이 그들에게 이전됩니다. 대개 인시던트 하나보다 저렴합니다.
- **패스키 또는 하드웨어 보안 키를 사용하십시오** — 지원하는 모든 계정에서. SMS 코드는 가로챌 수 있고, 비밀번호는 재사용되지만, 하드웨어 키는 피싱할 수 없습니다.
- **코드 호스트의 무료 보안 도구를 활성화하십시오.** GitHub의 Dependabot, 시크릿 스캔, CodeQL은 공개 저장소에서 무료이며 엔터프라이즈 도구가 잡는 것의 상당 부분을 포착합니다. 활성화는 몇 분 밖에 걸리지 않습니다.
- 오픈 소스 프로젝트를 유지 관리한다면, **SECURITY.md를 게시하십시오** — 연락처와 무엇을 기대할 수 있는지 명시합니다. AI 보조 스캔은 더 많은 보고서를 의미합니다. 명확한 접수 방법이 신호와 노이즈를 구분하는 데 도움이 되며, 선의의 보고자에게 그들의 노력이 헛되지 않을 것임을 알립니다.
