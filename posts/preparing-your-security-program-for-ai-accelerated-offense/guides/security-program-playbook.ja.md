# AI加速化された攻撃に備えるセキュリティプログラム・プレイブック

Anthropicのセキュリティエンジニアリング・研究チームが2026年4月10日に公開した記事 [Preparing your security program for AI-accelerated offense](https://claude.com/blog/preparing-your-security-program-for-ai-accelerated-offense) から抽出した、統合ロールアウトプレイブックです。この記事のガイダンスは既存の SOC 2 および ISO 27001 のコントロールに直接対応しています。内容はすべて原文に基づいており、新たに創作した内容はありません。

## 背景

AIは脆弱性が発見・悪用されるスピードを変えつつあります。同じ週の早い段階で、Anthropicは防御的サイバーセキュリティのために Claude Mythos Preview を活用する Project Glasswing を発表しました。今後24ヶ月以内に、これまで気づかれなかった膨大な数のバグ——場合によっては何年もの間——がAIモデルによって発見され、機能するエクスプロイトに連鎖させられるでしょう。AIツールを採用した防御側も、より速く動けるようになります。以下では「今すぐやるべきこと」の7つの領域と実践的な方法を示し、その後にレポートの品質と小規模チーム向けの簡略化についてのアドバイスを述べます。

## 1. パッチギャップを埋める

AIモデルは、パッチが当たっていないシステムにおける既知の脆弱性（すでにパッチ済みのもの）の特徴を認識することに非常に長けています。パッチを逆解析して機能するエクスプロイトに変換することは、これらのモデルが得意とする機械的な分析作業であり、パッチ公開からエクスプロイト登場までの窓は縮まっています。

- **CISA Known Exploited Vulnerabilities (KEV) catalog** にあるすべての項目を直ちにパッチ適用してください。このリストにある項目でネットワークから到達可能なものは緊急事態として扱うべきです。
- その他の優先順位付けには **EPSS**（Exploit Prediction Scoring System）を活用してください。CVEが今後30日以内に悪用される確率を毎日更新して提供します。
- インターネットに公開されているシステムのパッチ適用時間を短縮してください: エクスプロイトが公開されてから **24時間以内**、その他の脆弱性は数日以内。
- 自動更新による停止が許容可能なリスクである場合、パッチ展開と再起動を自動化してください。手動承認ステップは遅延を生み、**遅延が今や最大のリスクです。**

**実践的なヒント.** ほとんどのクラウドおよびOSベンダーはすでにパッチ自動化を提供しており、有効化は多くの場合、単純な設定変更で済みます。コンテナイメージと依存関係マニフェストについては、いくつかのオープンソーススキャナーが単一のCIステップとして実行され、CVEにKEVおよびEPSSデータを注釈として付加するため、優先順位付けが内蔵されています。

→ 再利用可能な [closing-the-patch-gap スキル](../skills/closing-the-patch-gap/SKILL.md) を参照してください。

## 2. 大幅に増加する脆弱性レポートへの対応準備

脆弱性管理プロセスは、ベンダーおよびアップストリームからのより多くのパッチに備えて計画を立てる必要があります。

- **発見量の桁違いな増加を見込んでください。** セキュリティ会議が依然としてスプレッドシートと週次ミーティング中心であれば、対応が追いつかなくなる可能性が高いです。
- オープンソースの依存関係のセキュリティを確認してください。**OpenSSF Scorecard** を使用すると、CIで実行され、メンテナンスされていないパッケージや不足しているシグナル（ブランチ保護、ファジングカバレッジ、署名済みリリース、メンテナーの活動状況）にフラグを立てます。
- サードパーティリスク管理を通じてベンダーにも同じ期待を適用してください: 加速されたエクスプロイトのタイムラインにどう備えているか、自社コードをスキャンしているかを確認してください。

**実践的なヒント.** 脆弱なコードの到達可能性を評価するツールを検討してください。迅速な展開への信頼を得るために、回帰テストを含めて新しい更新を継続的に提供する自動化プロセスを構築してください。

**脆弱性管理におけるAIの活用ポイント:**
- *トリアージの高速化.* フロンティアモデルは、重複した発見を除去し、資産情報から露出範囲を推定し、特定済みの影響を受けるコードパスとともに修復チケットの草案を作成できます。
- *依存関係の冗長性チェック.* 大規模なコードベースは、同じ役割を果たす複数のライブラリを蓄積する傾向があります（複数のHTTPクライアント、複数のJSONパーサーなど）。LLMにロックファイルを指し示し、どの依存関係が重複しているか（そして統合するとどうなるか）を尋ねることは、1時間の作業でしばしば成果を生みます。
- *アップグレードの自動化.* フロンティアモデルは、脆弱性レポートと並行してパッチを生成し、テストし、テストが通過することを検証しながらアップストリームのパッチを受け入れることができます。
- *AIベンダリング.* OpenSSF Scorecard のスコアが低く積極的にメンテナンスされていない小規模な依存関係については、実際に使用している機能だけをLLMに再実装させることを検討してください。

## 3. リリース前にバグを発見する

- **静的解析とAI補助コードレビュー**をCIに追加し、高信頼度の発見に対してマージをブロックしてください。誤検知がこれを非実用的にする場合は、ゲートではなくツールを改善してください。
- **OWASP ASVS** は、3段階の厳密さで「合格」の基準を定義しています。
- **自動化されたペネトレーションテスト**を継続的デリバリーに追加してください。攻撃者が本番環境に対して実行するのと同じスキャンをステージング環境に対しても実行してください。
- **ビルドパイプラインを保護してください。** **SLSA** フレームワークは段階的なパスを提供します。OpenSSFは SLSA Level 3 の証明を生成する再利用可能なGitHub Actionsワークフローを公開しており、SLSAの仕様が示唆するよりも大幅に少ない作業で実装できます。
- **Secure by Design** の実践を最低基準として採用してください（CISAの誓約: デフォルトMFA、デフォルトパスワードなし、透明な脆弱性報告）。
- 新しいコードには**メモリ安全な言語**（Rust、Go、マネージドランタイム）を優先してください。既存のC/C++コードは書き直す必要はありませんが、新しいC/C++コードには正当な理由が必要です。AI補助による書き直しはますます実用的な選択肢になっています。

**実践的なヒント.** OWASP Top 10 および言語固有のルールセットを持つSASTツールは、オープンソースとコードホスティングプラットフォームへの内蔵型の両方で幅広く利用可能です。GitHub の CodeQL が最も一般的な出発点です。

**ビルド時のAIの活用ポイント:**
- *AI脆弱性スキャン.* 攻撃者が使用するのと同種のモデルで、先に自社のコードとシステムをスキャンしてください。隔離されたエージェント、ノイズをフィルタリングするための検証ステップ、および既存のトリアージプロセスへのパスが必要です。**このセクションから一つだけ実装するなら、これを実装してください。** → [vulnerability-scanning-agent](../agents/vulnerability-scanning-agent.md) を参照してください。
- *パッチ生成.* フロンティアモデルは通常、SASTやスキャナーの発見に対してパッチを提案できます。開発者の仕事は「バグを理解して修正を書く」から「提案された修正が正しいか検証する」に移行し、後者の方が速いです。同じアプローチはメモリ安全な移行にも適用されます: 独立したCモジュールをテストと共にRustに移植し、同等性を検証します。

## 4. 既存コード内の脆弱性を発見する

長期間稼働している本番コードのほとんどは人間によって何度もレビューされていますが、フロンティアモデルによって検査されたことはありません。先を見越したスキャンにより、攻撃者が発見する前に現代のLLMの射程内にある脆弱性を特定します。

- **露出度順に優先順位を付けてください。** 信頼できない入力を解析したり、認証の決定を行ったり、インターネットから到達可能なコードから始めてください。
- **レガシーコードを含めてください。** 現在のレビュー慣行以前に作成されたコードは、最近のレビューが最も少なく、新たな検査から最も得るものが多いです。
- **修復のための予算を確保してください。** 古いコードのモデルスキャンは通常、SASTのロールアウトより発見件数が少ないですが、実際の脆弱性の割合が高いです。エンジニアリング時間を計画してください。

**実践的なヒント.** 現在の所有者が少ないインターネット公開サービスを1つ選び、その入力処理と認証ロジックをスキャンしてください。エージェントを隔離環境で実行し、検証ステップを追加し、確認された発見事項に対処してください。1つのサービスを適切に実施することで、より広範なプログラムのコストを見積もるための合理的な根拠が得られます。

## 5. 侵害を前提とした設計

摩擦（攻撃を退屈にすること）から価値を得る緩和策は、その退屈さを乗り越えられる攻撃者に対しては効果が薄れます。無制限の攻撃者の忍耐力に対しても機能するコントロールを優先してください。

- **ゼロトラストアーキテクチャを採用してください。** サービス間のすべてのリクエストをインターネットから来たものとして認証・認可してください。**CISA の Zero Trust Maturity Model** と **NCSC の zero trust principles** はどちらも段階的な採用パスを提供しています。
- **アクセスを資格情報ではなく検証済みハードウェアに紐付けてください。** 本番システムと機密性の高い内部ツールは、ハードウェアIDが証明された管理対象の従業員デバイスからのみアクセス可能であるべきで、**フィッシング耐性のある2FA（FIDO2 / パスキー）**と組み合わせて使用してください。盗まれた資格情報だけでは十分であってはなりません。サービス間の呼び出しもハードウェアIDに根ざすべきです。
- **サービスをIDによって分離してください。**
- **長期間有効なシークレットをIDプロバイダーが発行する短期・スコープ限定のトークンに置き換えてください。** 静的APIキーと埋め込まれた資格情報は、モデル補助のコード解析を持つ攻撃者が最初に見つけるものの一つです。

**実践的なヒント.** 完全なゼロトラストは数年単位のプログラムですが、**ID認識アクセスプロキシ**は内部サービスを再設計することなく、デバイス検証済みのMFAゲート付きアクセスを前面に配置します。主要なクラウドプロバイダーはすべて、ネイティブオプションとオープンソース/商用の代替品を提供しています。シークレットについては、最も広く共有されている資格情報を1つ管理対象のシークレットストアに移動してローテーションしてください。それが残りのための有効な強制関数になります。

## 6. 露出を削減してインベントリ化する

二つの原則: 把握していないものは守れない、そして露出する攻撃面が小さいほど攻撃対象も少なくなります。

- すべてのインターネット公開ホスト、サービス、APIエンドポイントの**現行インベントリ**を維持してください。ペンテストとレッドチーム演習にこれらを含めてください。
- **未使用システムを廃止してください。** 明確な所有者がいないレガシーサービスは通常、パッチも当たっていません。
- 各サービスが公開するものを**最小化してください。** ネットワークの受信はデフォルト拒否に; APIの攻撃面は実際に必要なものに限定してください。

**実践的なヒント.** インターネット全体のスキャンインデックスは公開検索可能です。自分のIPレンジとドメインを照会して、攻撃者の偵察が何を見ているかを確認してください。クラウド資産については、ネイティブのインベントリツール（**AWS Config、Azure Resource Graph、GCP Asset Inventory**）がすでに存在しています。作業はそれらを照会することです。

**露出に関するAIの活用ポイント:**
- *古いコードとシステムの整理.* コードベースとトラフィックログへの読み取りアクセスを持つモデルは、呼び出し元がなく、トラフィックも受けていないエンドポイントをリストアップし、それぞれを削除すると何に影響するかを説明できます。
- *自律的な外部レッドチーム.* 資格情報もソースアクセスも持たない状態で、AIオフェンシブエージェントを外部から自社の境界に向けてください。ソーススキャンでは見えないものを検出します: 忘れられたホスト、公開された管理インターフェース、デフォルト資格情報、誤って設定されたストレージ。インベントリ更新と同じサイクルで実行してください。→ [external-red-team-agent](../agents/external-red-team-agent.md) を参照してください。

## 7. インシデント対応時間を短縮する

エクスプロイトはパッチの数時間後に登場することがあります。日単位で計測される対応プロセスは遅すぎます。

- **アラートキューの先頭にモデルを配置してください。** すべての受信アラートは、人間が見る前に自動化された初期調査を受けます。読み取り専用のSIEMアクセスを持つ「トリアージエージェント」は、人間の判断が最も必要なアラートに注意を向けます。→ [security-triage-agent](../agents/security-triage-agent.md) を参照してください。
- **まず滞留時間とカバレッジを計測してください。** エクスプロイトの窓が縮まるときに最も重要であり、AI自動化が最も大きく動かせる2つの指標です。
- **インシデントに関する事務作業を自動化してください。** モデルがメモを取り、アーティファクトを収集し、並行した調査トラックを追求し、事後検討の草案を作成します。**人間**は封じ込め、開示、顧客コミュニケーションの判断を下します。人間の意思決定速度が証拠収集や文書作成によって制限されてはなりません。
- **検出のフライホイールをモデルが駆動させてください** ——脅威インテルの取り込み、検出候補の生成、ハンティング、発火のチューニング。
- **5件の同時インシデントを想定したテーブルトップ演習を実施してください。** AI能力を考慮すると、「月曜日の単一クリティカルCVE」という標準的な仮定は賢明でないかもしれません。同じ週に5件のインシデントでストレステストしてください。
- **MITRE ATT&CK に対して検出カバレッジをマッピングしてください。** 攻撃者のテクニックの標準的な語彙です。何を検出できて何を検出できないかを知ることは、一般的な「検出を改善する」という目標よりも有用です。横展開と資格情報アクセスのカバレッジを優先してください。
- **緊急変更手順を事前に確立してください。** 本番パッチに対する2週間の変更承認サイクル自体がセキュリティリスクです。封じ込めアクション（サービスをオフラインにする、資格情報をローテーションする、ネットワークパスをブロックする）も同様です。誰が承認し、どれほど速く行動するかを事前に決めてください。

**実践的なヒント.** 既知の高い誤検知率を持つノイジーなルールを1つ選んでください。フロンティアモデルを読み取り専用アクセスでそのアラートストリームに接続し、すべての発火に対して構造化された処分を生成させ、2週間にわたって人間のレビュアーとの一致を測定してください。一致が許容範囲であれば、次のルールに拡大してください。キュー全体を一度に自動化しようとしないでください。別途、**Atomic Red Team** はATT&Kの技術にマッピングされた小さく安全なテストのオープンソースライブラリです。いくつか実行し、既存のロギングが実際に何を検出したかを確認することは、具体的なカバレッジマップを生み出す半日の作業です。

## 脆弱性レポートの外部提出

コード（自社の依存関係、オープンソースプロジェクト、ベンダー製品）をスキャンしてアップストリームに報告する場合、レポートの品質が誰かが発見事項に対処するかどうかを決定します。オープンソースのメンテナーはすでに大量の低品質な自動レポートを受け取っており、多くはAIが生成したように見えるものを無視します。**レポートは人間が検証し、自分の名前を掲げる意志があるときにのみ送信すべきです。**

提出チェックリストと自己確認のための再利用可能な [writing-quality-vuln-reports スキル](../skills/writing-quality-vuln-reports/SKILL.md) を参照してください。

## セキュリティチームがない場合

上記のアドバイスのほとんどは専任のセキュリティ機能を想定しています。ソロ開発者、オープンソースメンテナー、小規模組織: リスクは同じく適用されますが、取るべきアクションはより単純です。

- **自動更新を有効にしてください** ——OS、ブラウザ、それを提供するすべてのアプリケーションに対して。継続的な努力を必要とせずに利用できる、単一の最も効果的なアクションです。
- **セルフホスティングよりマネージドサービスを優先してください。** セキュリティチームを持つプロバイダーにデータベース、認証、メールを運用させることで、パッチ適用の負担が移転します。通常、1回のインシデントよりも安価です。
- **パスキーまたはハードウェアセキュリティキーを使用してください** ——それをサポートするすべてのアカウントで。SMSコードは傍受可能で、パスワードは再利用されますが、ハードウェアキーはフィッシングできません。
- **コードホストの無料セキュリティツールを有効にしてください。** GitHubのDependabot、シークレットスキャン、CodeQLはパブリックリポジトリに対して無料で、エンタープライズツールが検出するものの相当部分を補足します。有効化は数分で完了します。
- オープンソースプロジェクトをメンテナンスしているなら、**SECURITY.md を公開してください** ——連絡先と期待できることを明記します。AI補助スキャンはより多くのレポートを意味します。明確な受け付け方法はシグナルとノイズを区別するのに役立ち、善意のレポーターに対して彼らの努力が無駄にならないことを示します。
