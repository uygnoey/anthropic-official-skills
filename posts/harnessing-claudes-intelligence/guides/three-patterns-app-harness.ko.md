# Claude의 지능을 따라잡는 앱 설계 3가지 패턴

이 가이드는 원문 글에서 제시한 3가지 패턴을 정리해, Claude의 역량이 빠르게 변해도 지연(latency)·비용(cost) 균형을 유지하며 앱(에이전트 포함)을 설계하는 데 참고할 수 있게 합니다.

## 패턴 1 — Claude가 이미 잘 아는 것으로 만들기
- Claude가 이미 잘 이해하는 도구를 우선 사용합니다.
- 글에서는 파일을 보고/만들고/수정할 수 있는 **bash 도구**와 **텍스트 편집 도구**를 강력한 기반으로 강조합니다.
- skills, programmatic tool calling, memory 같은 상위 구성도 이런 범용 도구 위에서 구축될 수 있습니다.

## 패턴 2 — “내가 이제 무엇을 그만해도 되지?”(하니스 가정 재검증)
에이전트 하니스는 종종 “Claude는 못 할 것”이라는 가정을 구조로 박아 넣습니다. Claude가 발전하면 그 구조가 오히려 병목(불필요한 무게)이 될 수 있습니다.

### Claude가 스스로 행동을 오케스트레이션하게 하기
- 모든 도구 결과를 Claude의 컨텍스트 윈도우로 흘려보내는 전제를 버립니다.
- 예를 들어 큰 테이블에서 한 컬럼만 필요해도 전체를 컨텍스트로 넣으면 느리고 비쌉니다.
- 대신 bash 또는 REPL 같은 코드 실행 도구를 제공해 Claude가 직접 코드를 작성하여
  - 도구 호출,
  - 출력 필터링/집계,
  - 다음 액션으로 파이프
  를 수행하게 하고, **최종적으로 필요한 최소 출력만** 컨텍스트에 반영되게 합니다.

### Claude가 스스로 컨텍스트를 관리하게 하기
- 드물게 쓰는 태스크 지시를 시스템 프롬프트에 잔뜩 미리 넣는 방식은 확장성이 떨어집니다.
- 점진적 공개(progressive disclosure)를 사용합니다.
  - 짧은 개요(예: skill의 YAML frontmatter)는 컨텍스트에 미리 들어가고,
  - 필요할 때만 Claude가 전체 내용을 읽습니다.
- 오래되거나 불필요해진 컨텍스트는 선택적으로 제거(컨텍스트 편집)합니다.
- 특정 작업을 격리해야 할 때는 subagent로 새 컨텍스트 창을 분기해 처리합니다.

### Claude가 스스로 컨텍스트를 ‘지속’하게 하기
- 장기 실행 에이전트는 단일 컨텍스트 윈도우 한계를 넘을 수 있습니다.
- compaction으로 과거 컨텍스트를 요약해 연속성을 유지합니다.
- memory folder를 사용하면 Claude가 중요한 정보를 파일에 기록하고 필요할 때 다시 읽을 수 있습니다.
- 글의 게임 예시는 ‘대사 전사’ 위주의 기록과, 실패에서 배운 ‘전술적 학습’을 디렉터리로 정리한 기록의 차이를 보여줍니다.

## 패턴 3 — 경계를 신중히 설정하기
하니스는 UX·비용·보안 경계를 강제하는 역할을 해야 합니다.

### 캐시 히트를 극대화하는 컨텍스트 설계
Messages API는 stateless이므로, 하니스는 매 턴 과거 컨텍스트를 함께 패키징합니다. 글에서는 프롬프트 캐시 재사용을 높이는 원칙을 제시합니다.
- **Static first, dynamic last**: 안정적인 내용(시스템 프롬프트, 도구)을 먼저, 자주 바뀌는 입력은 뒤에 둡니다.
- **Messages for updates**: 앞부분을 수정하지 말고 `system-reminder` 블록처럼 업데이트를 뒤에 덧붙입니다.
- **Don’t change models**: 캐시는 모델별로 분리되어 모델 변경은 캐시를 깨므로, 더 저렴한 모델이 필요하면 subagent를 사용합니다.
- **Carefully manage tools**: 도구 추가/삭제는 캐시를 무효화하므로, 동적 탐색은 tool search처럼 ‘덧붙이는’ 방식으로 처리합니다.
- **Update breakpoints**: 멀티턴 앱에서는 breakpoint를 최신 메시지로 이동해 캐시가 최신 prefix를 반영하도록 하며, auto-caching을 활용합니다.

### UX·관측성·보안을 위한 선언형(declarative) 도구 사용
- bash 도구는 범용성이 크지만, 하니스가 관찰하는 것은 ‘타입 없는 명령 문자열’뿐입니다.
- 행동을 타입이 있는 선언형 도구로 승격하면 하니스가
  - 액션을 가로채고 게이트(예: 외부 API 호출은 사용자 확인),
  - 안전장치 추가(예: edit 도구의 staleness check),
  - UX 표면화(모달, 선택지, 사용자 입력 대기),
  - 로깅/트레이싱/리플레이(구조화 인자)
  를 하기 쉬워집니다.
- 어떤 행동을 도구로 승격할지의 결정은 지속적으로 재평가해야 하며, 글은 Claude Code의 auto-mode가 bash 안전성을 ‘두 번째 Claude’로 판단하는 경계 패턴도 언급합니다.

## 지속 운영 관점
- Claude의 역량이 바뀔 때마다 가정을 다시 테스트합니다.
- 과거 모델 한계를 보완하려고 넣었던 리셋/스캐폴딩이 이제는 ‘죽은 무게’가 되었는지 점검하고 제거합니다.
