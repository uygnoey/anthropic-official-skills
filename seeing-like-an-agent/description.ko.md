## 이 글이 뭔가요
이 글은 Claude Code 팀이 모델의 관점에서 사고하는 방식(“에이전트처럼 보기”)으로 에이전트 도구를 설계·검증·개선하는 과정을 설명합니다.

## 언제 유용한가요
에이전트용 툴링(또는 에이전트 플랫폼)을 만들 때, 새 도구를 추가할지/기존 도구를 개선할지, 도구 인터페이스를 어떻게 설계할지, 모델에게 선택지를 과도하게 늘리지 않으면서도 필요한 기능을 제공하는 방법을 고민할 때 유용합니다.

## 핵심 포인트
- 도구 인터페이스는 모델의 현재 역량에 맞아야 하며, 모델 역량이 변하면 설계 가정도 재검토해야 합니다.
- 도구 추가에는 비용이 있습니다(선택지가 늘어 의사결정 부담 증가).
- 협소한 도구를 계속 늘리기보다, progressive disclosure(점진적 공개)와 재사용 가능한 스킬 문서를 우선 고려합니다.
- 도구가 병목이 되면 더 표현력이 높은 기본 요소로 교체합니다(예: todo 도구를 task 추상화로 대체).
- 사용자 확인(clarification)은 다른 도구에 억지로 끼워넣기보다 전용 질문 도구가 더 안정적일 수 있습니다.

## 번들 리소스
- Skill: 에이전트 도구 인터페이스 설계 휴리스틱, progressive disclosure, 새 도구 추가에 대한 “높은 기준”
