# AIによる攻撃の加速に備えたセキュリティプログラムの準備

## この記事について

AIモデルが脆弱性を素早く発見・連鎖・悪用する環境に対応するため、セキュリティプログラムをどのように適応させるかについて、Anthropicのセキュリティエンジニアリングおよびリサーチチームがまとめたガイドです。7つの「今すぐやること」領域（パッチギャップ、脆弱性レポートの件数増加、出荷前のバグ検出、既存コードの監査、侵害を前提とした設計、露出の低減、インシデントレスポンス）を中心に構成されており、高品質な脆弱性レポートの提出方法、個人または小規模チームのメンテナー向けアドバイス、そしてCISA KEV・EPSS・NIST SSDF・OWASP ASVS・SLSA・OpenSSF Scorecard・CISA Zero Trust Maturity Model・MITRE ATT&CK・NIST CSF 2.0などの標準規格へのマッピング参照表も含まれています。

## どんなときに役立つか

- AI主導の攻撃ペースに合わせて既存のセキュリティプログラムを再設計しているとき
- パッチギャップ管理・脆弱性管理・インシデントレスポンスのポリシー更新案を作成しているとき
- CI環境にAI支援レビュー（SASTに近い手法）を導入しており、責任ある展開手順が必要なとき
- AI支援による脆弱性レポートを提出または受領する予定があり、品質基準を確認したいとき
- 小規模な組織やオープンソースプロジェクトを率いており、簡潔なアクションリストが必要なとき

## 主なポイント

- パッチギャップを閉じる：KEVカタログは即時対応、その他はEPSSを基準に、インターネット公開システムは24時間以内を目標とし、安全な範囲で自動化する
- 脆弱性レポートが桁違いに増加することを想定し、AIでトリアージ・重複排除・パッチ案の作成を行う
- 出荷前にバグを検出する：CIにSAST＋AIレビューを組み込み、CDで自動ペネトレーションテストを実施し、ビルドチェーンを保護し（SLSA）、Secure by Designを採用し、メモリ安全な言語を優先する
- 既存コードを監査する：パーサー、認証境界、インターネットから到達可能なパス、レガシーコードを優先的に対象とする
- 侵害を前提に設計する：zero trust、ハードウェア紐づけのID、短命トークン、「摩擦」だけのコントロールを実質的な障壁に置き換える
- 露出を低減・棚卸しする；オーナー不在のサービスを廃止する；自律型外部レッドチームを運用する
- インシデントレスポンスを短縮する：キューの先頭にトリアージエージェントを置き、滞留時間の指標を設け、AIをスクライブとして活用し、緊急変更手順を事前に合意しておく
- AI支援の脆弱性レポートは、人間が検証し、自分の名前で提出できるものだけを送る
- すべてが既存のSOC 2 / ISO 27001コントロールに自然にマッピングされる
