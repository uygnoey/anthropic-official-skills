# セッション管理と1Mコンテキストウィンドウ

*Using Claude Code: session management and 1M context*(2026年4月15日)の内容を読み物として整理しました。

## なぜ重要か

Claude Codeチームが `/usage` スラッシュコマンドを公開し、顧客との会話で一貫したテーマが浮かび上がりました。使い方の幅が非常に大きいということです。ターミナルを何日も開いたままにする開発者もいれば、プロンプトごとに新しいセッションを始める人もいます。Claude Code のコンテキストが100万トークンまで拡張された今、この選択はますます重要になっています。

結果の差のほとんどは **コンテキストウィンドウをどう管理するか** に行き着きます。

## 基本の整理

コンテキストウィンドウは、モデルが次の応答を生成するときに一度に「見える」すべてです。システムプロンプト、ここまでの会話、あらゆるツール呼び出しと出力、そして読み込んだファイル全部です。Claude Code のウィンドウは100万トークンまで保持します。

しかしコンテキストにはコストがあります。**context rot** と呼ばれる現象です。コンテキストが大きくなるほど注意が多くのトークンに分散し、古い無関係な内容が現在のタスクを邪魔してモデル性能が劣化します。

ウィンドウが上限に近づくと、Claude Code はタスクを短い要約にまとめて新しいウィンドウで作業を継続します。これが **コンパクション(compaction)** です。自分でも発火させられます。

## 各ターンは分岐点

Claude の応答が終わるたびに5つの選択肢があります:

- **Continue** — 同じセッションで次のメッセージを送る。
- **`/rewind`**(または Esc 2回)— 過去のメッセージに戻ってそこから再プロンプト。以降は捨てられる。
- **`/clear`** — 新しいセッションを始める。普通は、いま学んだ内容をブリーフとして一緒に渡す。
- **`/compact`** — これまでのセッションを要約し、その要約の上に作業を続ける。
- **サブエージェント** — 次の作業を別のクリーンなコンテキストを持つエージェントに委譲し、結果だけ親に戻す。

Continue が最も自然ですが、残り4つはコンテキストを意図的に管理するためにあります。

## いつ新しいセッションにするか

原則:**新しいタスク ⇒ 新しいセッション**。1Mウィンドウのおかげでフルスタックアプリをゼロから作るような長いタスクも安定してできるようになりましたが、context rot は依然として起きます。

例外もあります。実装したばかりの機能のドキュメントを書くのは厳密には新しいタスクですが、ファイルを再読込させる時間とコストを払いたくないでしょう。セッションは維持しましょう。

## 修正より巻き戻し

`/rewind`(Esc 2回)は任意の過去メッセージに戻り、そこから再プロンプトします。以降はすべて破棄されます。

Claudeが5つのファイルを読んであるアプローチを試し、うまくいかなかったとき、本能的には「うまくいかなかった、Xを試して」と返したくなります。しかし、有用なファイル読み込み直後まで巻き戻して、学んだ内容を込めて再プロンプトする方がしばしば良い選択です。*「アプローチAは使わない。foo モジュールはそれを公開していない。最初からBに行け。」*

「ここから要約して」や `/rewind` を使い、Claude に巻き戻し前に引き継ぎメモを書かせることもできます。試してダメだったと未来の自分に伝えるメッセージだと思ってください。

## Compact と Clear

`/compact` はモデルに会話を要約させ、履歴をその要約に置き換えます。情報は失われますが、手間はかからず、Claude が学習や重要ファイルを丁寧にまとめてくれることもあります。指示で方向付けも可能です:`/compact focus on the auth refactor, drop the test debugging`。

`/clear` はまっさらから始めますが、重要なことを**あなたが**書き出さねばなりません。*「auth ミドルウェアのリファクタ中、制約は X、関連ファイルは A と B、アプローチ Y は除外済み。」* 手間は増えますが、持ち越される内容を完全にコントロールできます。

## 自動コンパクションが失敗する理由

悪い自動コンパクションは通常、モデルがあなたの次の方向を予測できなかったときに起きます。長いデバッグセッションがコンパクションされ、次のメッセージが「じゃあ `bar.ts` で見たもう一つの警告も直して」だったとします。セッションがデバッグに集中していたため、その警告は要約から落ちているかもしれません。

ここが厄介なのは、モデルがコンパクションする瞬間は最も能力が落ちているタイミングだからです。context rot がすでに効いています。1Mコンテキストのおかげで、ウィンドウが埋まる前にヒント付きで先回りして `/compact` する余裕が生まれました。

## サブエージェントと新鮮なコンテキスト

サブエージェントは、中間出力が大量で結論しか要らない作業で真価を発揮します。

Claude が Agent ツールでサブエージェントを起動すると、そのサブエージェントは自前のクリーンなウィンドウを得て必要な作業をこなし、結果を合成して最終レポートだけを戻します。親は中間ノイズを見ません。

Anthropic の判断基準:*そのツール出力をまた使うのか、結論だけで十分か?*

サブエージェントを明示的に呼ぶプロンプト例:

- 「次のスペックファイルを基にこの作業の結果を検証するサブエージェントを立ち上げて。」
- 「別のコードベースを読んで auth フローがどう実装されているかを要約するサブエージェントを立ち上げ、そのやり方で自分で実装して。」
- 「私の git 変更履歴を基にこの機能のドキュメントを書くサブエージェントを立ち上げて。」

## 決定表

| 状況 | 使うべき手段 | 理由 |
|---|---|---|
| 同じタスクでコンテキストがまだ有効 | Continue | ウィンドウの中身はまだ役立つ。作り直す無駄を省く。 |
| Claudeが誤った道に進んだ | Rewind(Esc 2回)| 有用なファイル読み込みを残し、失敗だけ捨てて学びで再プロンプト。 |
| 作業途中だがセッションが古いデバッグ/探索で肥大化 | `/compact <hint>` | 低コスト。Claude が重要事項を判断。必要なら指示で誘導。 |
| 本当に新しいタスクを始める | `/clear` | rot ゼロ。何を持ち越すかを完全にコントロール。 |
| 次の工程で結論しか必要ない大量出力が発生 | サブエージェント | 中間ノイズは子のコンテキストに残し、結果だけ親に戻る。 |
