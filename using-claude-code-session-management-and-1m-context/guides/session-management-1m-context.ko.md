# 세션 관리와 1M 컨텍스트 윈도

*Using Claude Code: session management and 1M context* (2026년 4월 15일)의 지침을 서술형으로 정리했습니다.

## 왜 중요한가

Claude Code 팀이 `/usage` 슬래시 명령을 출시하면서 고객들과 나눈 대화에서 한 가지 공통된 주제가 발견됐습니다. 사용 패턴이 사람마다 크게 다르다는 점입니다. 어떤 개발자는 터미널 하나에 세션을 며칠씩 열어 두고, 어떤 사람은 프롬프트마다 새 세션을 엽니다. Claude Code의 컨텍스트가 100만 토큰으로 확장되면서 이 선택은 덜 중요해지기는커녕 더 중요해졌습니다.

결과 차이는 거의 전부 **컨텍스트 윈도를 어떻게 관리하느냐**에서 옵니다.

## 기본 개념

컨텍스트 윈도는 모델이 다음 응답을 생성할 때 한 번에 볼 수 있는 모든 것을 가리킵니다. 시스템 프롬프트, 지금까지의 대화, 모든 도구 호출과 출력, 읽은 파일 전부입니다. Claude Code의 윈도는 100만 토큰까지 담습니다.

그러나 컨텍스트에는 대가가 따릅니다. 바로 **context rot** 현상입니다. 컨텍스트가 커질수록 어텐션이 더 많은 토큰으로 분산되고, 오래된 관련 없는 내용이 현재 작업을 방해하면서 성능이 저하됩니다.

윈도가 한계에 가까워지면 Claude Code는 작업을 요약해 짧은 설명으로 대체하고 새 윈도에서 작업을 이어갑니다. 이것이 **컴팩션(compaction)** 입니다. 직접 트리거할 수도 있습니다.

## 매 턴은 분기점이다

Claude의 응답이 끝날 때마다 다섯 가지 선택지가 있습니다:

- **Continue** — 같은 세션에서 다음 메시지를 보냅니다.
- **`/rewind`** (또는 Esc 두 번) — 이전 메시지로 돌아가 거기서 다시 프롬프트합니다. 그 이후 메시지는 버려집니다.
- **`/clear`** — 새 세션을 시작합니다. 보통 방금 배운 내용을 요약한 브리핑을 함께 넣습니다.
- **`/compact`** — 지금까지 세션을 요약하고 그 요약 위에서 작업을 계속합니다.
- **서브에이전트** — 다음 작업 덩어리를 깨끗한 컨텍스트의 에이전트에게 위임하고 결과만 부모 컨텍스트로 가져옵니다.

Continue가 기본값이지만 나머지 네 가지가 컨텍스트를 의도적으로 관리하는 방법입니다.

## 언제 새 세션을 시작할까

원칙: **새 작업 = 새 세션**. 1M 윈도 덕분에 전체 스택 앱을 바닥부터 만드는 등 더 긴 작업을 안정적으로 할 수 있지만, context rot는 여전히 일어납니다.

예외도 있습니다. 방금 구현한 기능의 문서를 쓰는 일은 엄밀히는 새 작업이지만, 그 파일들을 다시 읽게 하면 느리고 비쌉니다. 세션을 유지하세요.

## 수정 대신 되돌리기

`/rewind`(Esc 두 번)는 임의의 이전 메시지로 돌아가서 거기서부터 다시 프롬프트합니다. 그 이후의 모든 내용은 사라집니다.

Claude가 파일 다섯 개를 읽고 어떤 접근을 시도했다가 실패했을 때, 본능적으로 "그건 안 됐어, X로 해봐"라고 입력하기 쉽습니다. 더 나은 방법은 파일 읽기 직후로 되돌려서 배운 내용을 담아 다시 프롬프트하는 것입니다. *"접근 A는 쓰지 마. foo 모듈이 그걸 노출하지 않아. 바로 B로 가."*

"여기서부터 요약해 줘" 또는 `/rewind`를 써서 Claude가 인수인계 메모를 남기게 할 수도 있습니다. 뭔가 시도해 보고 안 됐다는 사실을 미래의 자신에게 알려주는 메시지라고 생각하면 됩니다.

## Compact vs Clear

`/compact`는 모델에게 대화를 요약하라고 하고 히스토리를 그 요약으로 대체합니다. 손실이 있지만 당신이 직접 할 일은 없고, Claude가 중요한 학습과 파일을 꼼꼼히 담아줄 수도 있습니다. 지시를 덧붙일 수도 있습니다: `/compact focus on the auth refactor, drop the test debugging`.

`/clear`는 완전히 새로 시작하는 대신 **당신이** 무엇이 중요한지 적어야 합니다. *"우리는 auth 미들웨어를 리팩토링 중이고, 제약은 X, 중요한 파일은 A와 B이며, 접근 Y는 배제했다."* 수작업이 더 들지만 이월되는 컨텍스트를 완전히 통제할 수 있습니다.

## 나쁜 자동 컴팩션의 이유

나쁜 자동 컴팩션은 대개 모델이 당신의 다음 방향을 예측하지 못했을 때 일어납니다. 긴 디버깅 세션이 컴팩션되고 나서 다음 메시지로 "이제 `bar.ts`에서 봤던 다른 경고도 고쳐줘"라고 했을 때, 세션이 디버깅에 집중돼 있었기 때문에 그 경고가 요약에서 빠졌을 수 있습니다.

이 시점이 특히 까다로운 이유는 모델이 컴팩션할 때 가장 성능이 떨어지기 때문입니다 — 이미 context rot이 진행된 상태입니다. 1M 컨텍스트 덕분에 이제 윈도가 꽉 차기 전에 다음 방향을 힌트로 담아 선제적으로 `/compact`할 여유가 생겼습니다.

## 서브에이전트와 깨끗한 컨텍스트

서브에이전트는 중간 출력이 많지만 결론만 필요할 때 빛을 발합니다.

Claude가 Agent 도구로 서브에이전트를 띄우면, 서브에이전트는 깨끗한 컨텍스트 윈도를 받아 필요한 작업을 하고 결과를 합성해 최종 보고서만 반환합니다. 부모는 중간 잡음을 보지 않습니다.

Anthropic의 판단 기준: *도구 출력을 또 쓸 건가, 결론만 필요한가?*

명시적으로 서브에이전트를 호출하는 프롬프트 예시:

- "다음 스펙 파일을 기준으로 이번 작업 결과를 검증하도록 서브에이전트를 띄워 줘."
- "다른 코드베이스를 훑어서 auth 흐름이 어떻게 구현돼 있는지 요약하도록 서브에이전트를 만들고, 그 방식 그대로 여기서 구현해."
- "내 git 변경사항을 기반으로 이 기능의 문서를 쓰도록 서브에이전트를 띄워."

## 한눈에 보는 결정표

| 상황 | 고려할 도구 | 이유 |
|---|---|---|
| 같은 작업이고 컨텍스트가 여전히 유효 | Continue | 윈도에 있는 게 아직 쓸모 있음. 다시 만들 비용 낭비 |
| Claude가 잘못된 방향으로 감 | Rewind (Esc 두 번) | 유용한 파일 읽기는 유지하고 실패한 시도만 버리고 다시 프롬프트 |
| 작업 중이지만 세션이 낡은 디버깅/탐색으로 비대 | `/compact <hint>` | 수고 적음. Claude가 중요한 것을 판단하되 지시로 방향 설정 가능 |
| 진짜로 새 작업 시작 | `/clear` | rot 없음. 이월할 내용을 완전히 통제 |
| 다음 단계가 결론만 필요한 대량 출력 생성 | 서브에이전트 | 중간 잡음은 자식 컨텍스트에 두고 결과만 부모로 |
